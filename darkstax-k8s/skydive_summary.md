
Skydive-UI provides a hierarchical, graph-based visualization of Kubernetes environments that unifies structural containment, logical relationships, and runtime status into a single interactive topology view. The Kubernetes topology is rendered as a graph rooted at the cluster level and expands progressively into namespaces, workloads, and runtime components, allowing users to visually explore Kubernetes at multiple levels of abstraction.

At the core of the view is a true hierarchical model of Kubernetes resources. The topology starts at the cluster and expands into namespaces, which in turn contain Kubernetes objects such as deployments, replica sets, daemon sets, stateful sets, services, ingresses, network policies, jobs, cron jobs, config maps, secrets, and persistent volume claims. Pods appear under their owning controllers or namespaces and can be expanded further to reveal individual containers. Cluster-scoped objects such as nodes, persistent volumes, and storage classes are also represented, enabling a unified view of both application and infrastructure layers.

Beyond simple containment, Skydive-UI models Kubernetes as a graph with multiple relationship types. Ownership edges represent parent–child hierarchy (for example namespace → pod or pod → container), while relationship edges represent how resources interact. These include service-to-pod selector relationships, network policy enforcement relationships, storage dependencies between PVCs and PVs, and other declarative Kubernetes links that are not visible in a traditional tree view. This allows users to understand not only where a resource lives, but how it influences and depends on other resources.

The topology supports interactive expand and collapse navigation, enabling users to progressively drill down from cluster-level views to fine-grained container detail without overwhelming the screen. Large clusters can be explored incrementally, focusing only on relevant namespaces or workloads. This interaction model is consistent across all Kubernetes resource types and supports exploratory analysis and troubleshooting workflows.

Skydive-UI includes filtering and highlighting capabilities driven by Gremlin-style queries. Users can reduce complex clusters to an application slice by filtering on labels, namespaces, or resource types, or highlight dependency neighborhoods such as all pods affected by a given network policy or secret. This enables fast isolation of ownership boundaries, dependency chains, and blast radius analysis.

Each Kubernetes object in the hierarchy is status-aware. Nodes in the graph are visually marked as healthy or unhealthy using color indicators derived from Kubernetes status metadata. This allows users to quickly spot failing pods, unbound volumes, or unavailable resources and visually trace upstream and downstream dependencies to identify root causes, such as a pod failing due to a broken storage or policy dependency.

Selecting any node in the topology reveals rich metadata in the UI, including resource type, name, namespace, labels, and status, along with access to the full underlying Kubernetes object definition captured in the topology. This enables deep inspection of configuration and runtime state without leaving the topology view or switching tools.

The Kubernetes hierarchy can be rendered from live Skydive analyzer data or from a static topology export such as the provided `kubernetes.json` demo. This allows offline exploration, reproducible debugging scenarios, demos, and UI development without a running Kubernetes cluster. The static topology behaves identically to live data from a user interaction perspective.

Overall, Skydive-UI’s Kubernetes hierarchical topology view delivers a unified, interactive graph that combines structural hierarchy, logical relationships, runtime status, and rich metadata. It is designed to help users understand complex Kubernetes environments, analyze dependencies, diagnose failures, and explore application and infrastructure interactions in a single, visual context.
